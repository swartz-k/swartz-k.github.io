AI Model Server - Tensorflow#

Tensorflow 官方提供了 serving 仓库，我们可以使用 docker 在文档提示下做一个 MVP 验证，参考文档。

# Download the TensorFlow Serving Docker image and repo
docker pull tensorflow/serving

git clone https://github.com/tensorflow/serving
# 测试模型父目录
TESTDATA="$(pwd)/serving/tensorflow_serving/servables/tensorflow/testdata"

# 挂载模型 saved_model_half_plus_two_cpu 至路径 /models 下，启动serving
docker run -t --rm -p 8501:8501 \
    -v "$TESTDATA/saved_model_half_plus_two_cpu:/models/half_plus_two" \
    -e MODEL_NAME=half \
    tensorflow/serving

到这里我们可以看到基本的 serving 使用方法。

将模型目录挂载 并通过环境变量指定模型名称， 启动 tensorflow/serving 镜像即可。

那么这个docker 是怎么把我们的 tensorflow 模型转换成可以访问的RESTFul API呢，来看一下 Dockerfile 核心构建内容。

# 制定版本的构建镜像
ARG TF_SERVING_VERSION=latest
ARG TF_SERVING_BUILD_IMAGE=tensorflow/serving:${TF_SERVING_VERSION}-devel

FROM ${TF_SERVING_BUILD_IMAGE} as build_image

# 从构建镜像拷贝二进制 tensorflow_model_server
COPY --from=build_image /usr/local/bin/tensorflow_model_server /usr/bin/tensorflow_model_server

# 根据环境变量 MODEL_NAME、MODEL_BASE_PATH 生成 entrypoint.sh
RUN echo '#!/bin/bash \n\n\
tensorflow_model_server --port=8500 --rest_api_port=8501 \
--model_name=${MODEL_NAME} --model_base_path=${MODEL_BASE_PATH}/${MODEL_NAME} \
"$@"' > /usr/bin/tf_serving_entrypoint.sh \
&& chmod +x /usr/bin/tf_serving_entrypoint.sh

ENTRYPOINT ["/usr/bin/tf_serving_entrypoint.sh"]

我们可以看到核心是使用 tensorflow_model_server 进行模型的 serving。那么 tensorflow_model_server 具体逻辑是怎样的呢？（关于如何生成，可以参考这个 Dockerfile ）

具体代码在 tensorflow_serving/model_servers，从 main.c 里可以看到提供了很多参数。核心代码如下。

 tensorflow::serving::main::Server server;
  const auto& status = server.BuildAndStart(options);
  if (!status.ok()) {
    std::cout << "Failed to start server. Error: " << status << "\n";
    return -1;
  }
  server.WaitForTermination();

那么我们接下来看一下 BuildAndStart 是做了什么。

Status Server::BuildAndStart(const Options& server_options) {
  // 依据 server_options 检查各种前置条件 && 初始化
  // 生成 model_service_、prediction_service_
  model_service_ = absl::make_unique<ModelServiceImpl>(server_core_.get());
  prediction_service_ = absl::make_unique<PredictionServiceImpl>(predict_server_options);

  // 生成 grpc_server_、http_server_
  ::grpc::ServerBuilder builder;
  builder.RegisterService(model_service_.get());
  builder.RegisterService(prediction_service_.get());
  grpc_server_ = builder.BuildAndStart();

  http_server_ = CreateAndStartHttpServer(
          server_options.http_port, server_options.http_num_threads,
          server_options.http_timeout_in_ms, monitoring_config,
          server_core_.get());

  return Status::OK();
}

我们可以看到核心是构建 grpc_server_ 和 http_server_ 用来对外提供 model_service_ 和 prediction_service_ 。这里可能会产生误解，其实model_service_ 只是对 server_core_ 的一层封装，负责模型状态和模型配置 ModelServerConfig 的管理。 prediction_service_ 的核心也是 server_core_ , 但是会负责模型实际的元数据信息查询、预测等服务。

grpc_server_ 是标准库的用法，不过多介绍，可以参考 官方资料 。

http_server_ 核心是提供 http 服务和用于监控的 Prometheus Handler，实现位于 tensorflow_serving/util/net_http/server/internal/evhttp_server.cc 提供 http 服务的逻辑比较简单，核心代码如下：

auto server = net_http::CreateEvHTTPServer(std::move(options));
std::shared_ptr<RestApiRequestDispatcher> dispatcher =
      std::make_shared<RestApiRequestDispatcher>(timeout_in_ms, core); // 这里的core 为 上面 server_core_.get()
// 注册 dispatcher server->RegisterRequestDispatcher

关于 server_core_ 本质是由 AspiredVersionsManager 包装而成，ServerCore 负责对多版本的模型进行管理，

// 创建ServerCore
Status ServerCore::Create(Options options,
                          std::unique_ptr<ServerCore>* server_core) {
  auto model_server_config = options.model_server_config;
  server_core->reset(new ServerCore(std::move(options)));
  TF_RETURN_IF_ERROR(
      (*server_core)->Initialize(std::move(aspired_version_policy)));
  return (*server_core)->ReloadConfig(model_server_config);
}

// 核心逻辑：创建Adapter
Status ServerCore::CreateAdapter(
    const string& model_platform,
    std::unique_ptr<StoragePathSourceAdapter>* adapter) const {
  auto config_it =
      options_.platform_config_map.platform_configs().find(model_platform);

  const tensorflow::Status status =
      StoragePathSourceAdapterRegistry::CreateFromAny(adapter_config, adapter);
}

这里初始化逻辑并不负责，比较重要的是 ReloadConfig，通过对 AddModelsViaModelConfigList -> CreateAdapters -> CreateAdapter 一些列函数调用，创建 StoragePathSourceAdapter。

创建逻辑里的 Options 有几个重要的参数需要特别强调。

  • ModelServerConfig指定要加载的模型。模型是通过model_config_list声明的一个静态模型列表，或者通过声明custom_model_config，它定义了一种自定义方式来声明可能在运行时更新的模型列表。
  • PlatformConfigMap从平台名称（例如tensorflow）映射 到PlatformConfig用于创建 SourceAdapter. SourceAdapter 提供 StoragePath（发现模型版本的路径）到模型Loader（从存储路径加载模型版本并提供状态转换接口Manager）的转换。同时如果提供了 SavedModelBundleSourceAdapterConfig 这里会创建一个 SavedModelBundle。

SavedModelBundle是 TensorFlow Serving 的关键组件。它代表从给定路径加载的 TensorFlow 模型，并提供与Session::Run 相同的接口来运行推理。

有了所有这些，ServerCore内部处理逻辑如下：

  • 由 FileSystemStoragePathSource监控在 model_config_list 中声明的模型路径。
  • SourceAdapter使用中 PlatformConfigMap 声明的模型平台实例化，并连接到 FileSystemStoragePathSource。只要在导出路径下发现新模型版本，就会由 SavedModelBundleSourceAdapter 将其转换为为Loader<SavedModelBundle>.
  • 由 Manager 管理 SavedModelBundleSourceAdapter 创建的 Loader. ServerCore暴露 Manager 接口。每当有新版本可用时，就会由AspiredVersionsManager` 加载新版本，卸载旧版本（默认行为可配置）。
